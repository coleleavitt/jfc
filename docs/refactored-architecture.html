JFC Refactored Architecture

What's redundant, what to unify, and the clean target state

🔴 Current Redundancy
3 Parallel Agent Tracking Systems
• BackgroundTask (engine/app.rs) — in-memory, UI state
• BackgroundAgentInfo (daemon/state.rs) — persistent, daemon state
• InProcessTeammateState (swarm/types.rs) — teammate-specific

All track the same thing: status, tokens, tool_use_count, error, timestamps.
3 Status Enums
• AgentStatus {Running, Completed, Failed}
• BackgroundAgentStatus {Pending, Running, Completed, Failed, Cancelled}
• TeammateStatus {Running, Idle, Completed, Failed, Killed}

Idle/Killed are teammate-specific; rest are identical.
2 Spawn Config Types
• AgentConfig (background.rs) — task_description, max_tokens
• TeammateRunnerConfig (swarm/runner.rs) — identity, prompt, model, provider

TeammateRunnerConfig is a superset.
2 Mailbox Systems
• swarm/mailbox.rs — file-based JSON inbox per agent
• SendMessage tool writes to mailbox
• TeammateEvent::MessageSent goes through events

Economy's SwarmProvider::send_message() is a stub that logs. The daemon doesn't use mailbox at all — it uses log files.
2 Task List Systems
• TaskStore (jfc-session) — shared task list for factory/teams
• ScheduledTaskRegistry (jfc-daemon) — cron/scheduled tasks

Different concerns but similar CRUD + lifecycle patterns. Both have their own persistence, their own status enums, their own files.
Scattered Agent Identity
• AgentId(u64) in background.rs
• AgentId(String) in jfc-economy/types.rs
• agent_id: String in swarm TeammateIdentity
• task_id: String in BackgroundTask

No unified identity type.
🟢 Proposed Unified Model
Single AgentState struct
pub struct AgentState {
    pub id: AgentId,
    pub status: AgentStatus,           // Running | Idle | Completed | Failed | Cancelled
    pub role: AgentRole,               // Solo | Teammate | Solver | Validator | Council
    
    // Progress (all agents share this)
    pub token_count: u64,
    pub tool_use_count: u32,
    pub last_tool: Option<String>,
    
    // Lifecycle
    pub started_at: SystemTime,
    pub completed_at: Option<SystemTime>,
    pub error: Option<String>,
    
    // Backend-specific (Option fields, only populated for relevant roles)
    pub team_name: Option<String>,     // Teammate
    pub worktree_path: Option<PathBuf>, // Solver
    pub trust_score: Option<i8>,       // Economy agents
    pub idle_reason: Option<String>,   // Teammate
}
Single AgentRegistry trait
pub trait AgentRegistry: Send + Sync {
    fn spawn(&self, config: SpawnConfig) -> AgentId;
    fn status(&self, id: &AgentId) -> Option<AgentStatus>;
    fn state(&self, id: &AgentId) -> Option<AgentState>;
    fn abort(&self, id: &AgentId) -> bool;
    fn complete(&self, id: &AgentId, result: AgentResult);
    fn list(&self) -> Vec<AgentState>;
}

Economy Integration
Current: Isolated bounded context
• jfc-economy has its own AgentId, SolverPool, ValidationPool
• SwarmProvider trait is implemented as a stub in tools/economy.rs
• Solvers/validators don't show up in the TUI's background task panel

Unified: Economy as AgentRole
// Solver spawns through the same AgentRegistry
let solver_id = registry.spawn(SpawnConfig {
    role: AgentRole::Solver { bounty_id, worktree },
    prompt: solver_prompt,
    ..
});

// Registry tracks it like any other agent
// TUI can show solver progress in the same panel
// Trust score is just another field on AgentState

// Economy orchestrator becomes a coordinator, not an owner
impl BountyCoordinator {
    fn run_cycle(&self, bounty_id: &str, registry: &dyn AgentRegistry) {
        let solver_ids = (0..n_solvers).map(|i| {
            registry.spawn(solver_config(i))
        }).collect();
        
        // Wait for all to complete
        for id in &solver_ids {
            while !registry.status(id).is_terminal() {
                tokio::time::sleep(Duration::from_millis(100)).await;
            }
        }
        
        // Settlement uses unified state
        let solutions = solver_ids.iter()
            .filter_map(|id| registry.state(id))
            .filter(|s| s.status == Completed)
            .collect();
    }
}

Clean Dispatch (After Refactor)
// tools/dispatch.rs becomes trivial routing
match (kind, input) {
    // Solo tools — unchanged
    (Bash, ..) => execute_bash(..),
    (Read, ..) => execute_read(..),
    
    // All agent spawns go through registry
    (Task, TaskInput { run_in_background: true, .. }) => {
        let id = registry.spawn(SpawnConfig::from_task_input(input));
        ExecutionResult::success(format!("Spawned agent {id}"))
    }
    
    // SendMessage uses MessageBus
    (SendMessage, SendMessageInput { to, text, .. }) => {
        let to_id = registry.resolve_name(&to)?;
        message_bus.send(&to_id, Message::new(text));
        ExecutionResult::success("Message sent")
    }
    
    // Economy tools spawn through same registry
    (PostBounty, ..) => bounty_coordinator.post(..),
    (RunBounty, ..) => bounty_coordinator.run(registry, ..).await,
    
    // Council is just parallel spawn + join
    (Council, ..) => {
        let ids: Vec<_> = models.iter().map(|m| {
            registry.spawn(SpawnConfig { role: Council, model: m, .. })
        }).collect();
        let results = join_all(ids.iter().map(|id| registry.wait(id))).await;
        // Arbiter synthesis
    }
}

What Stays the Same
These are NOT redundant — keep them
• TaskStore (jfc-session) — this is the shared TODO list, not agent tracking
• ScheduledTaskRegistry (jfc-daemon) — cron jobs, separate concern
• TeamFile (swarm/types.rs) — team roster config, not agent state
• Charter (jfc-economy) — rules/policy, not agent tracking
• TokenLedger (jfc-economy) — accounting, separate concern
• Provider trait — model API abstraction, orthogonal
• stream.rs loop — streaming logic, uses registry but doesn't duplicate it
