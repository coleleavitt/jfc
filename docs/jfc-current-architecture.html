Fragmented Architecture
As-Is Analysis

Three AgentIds.
Four status enums. Zero shared state.

The current JFC runtime has one conceptual domain — an invoked agent worker — expressed through a tangle of overlapping types, disconnected lifecycle authorities, and ad-hoc routing logic baked into the dispatch layer. This document is a full read of the problem before the refactor.

Confirmed UI Bug — Dual State Authority
The Agents modal rendered the team section from TeamContext.teammates.abort_tx while the canonical row above rendered from BackgroundTask.status. A completed teammate could therefore show as Completed in one row and Running in another — because two sources of truth drifted apart. This is a structural problem, not a fixable edge case.

4 Disconnected Messaging Channels
Teammate Mailbox

swarm/mailbox.rs writes JSON to ~/.claude/teams/{name}/inboxes/{agent}.json. Works cross-process but is slow (file poll) and not used by daemons.

TeammateEvent Channel

TeammateEvent::MessageSent flows through a tokio mpsc channel for in-process teammates. Separate from the file mailbox — same message concept, different path.

BackgroundAgentCompletion

Daemon workers signal completion through BackgroundAgentCompletion events that get polled by sync_detached_background_tasks(). Yet another message format.

SwarmProvider Stub

SwarmProvider::send_message() in tools/economy.rs is a stub that only logs. Economy agents cannot actually send messages — the channel is dead.

Current State vs. Target
Current — Fragmented
3 different AgentId types across crates — no cross-system correlation
4 status enums with overlapping but slightly different variants
BackgroundTask and TeamContext both own teammate status — they drift
Economy solvers and council members are invisible to the agent roster UI
4 messaging channels, one of which is a no-op stub
dispatch_heavy.rs holds global mutexes for council and bounty — not composable
Team membership checked via team_name string field deep inside dispatch — not typed
Target — Unified (jfc-agent crate)
One AgentId(Uuid) type used everywhere — daemon, swarm, economy, council
One AgentStatus enum with all meaningful variants (Idle included for teammates)
AgentRegistry trait is the single source of truth — UI reads only from here
Solvers spawn through registry — they appear in the roster like any other agent
MessageBus unifies in-memory mpsc and file-backed inboxes behind one API
dispatch.rs becomes trivial routing — no mutexes, no ad-hoc splitting
Agent role is a typed AgentRole enum on SpawnConfig — not a string check
