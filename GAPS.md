## Current read of the gap

JFC already has the core fanout primitive people usually associate with Claude Code workflows: the JS workflow runner exposes `agent()`, `parallel()`, `pipeline()`, `workflow()`, `phase()`, and `log()`; gates concurrent agents at `min(16, cpus - 2)`; and hard-caps workflow lifetime dispatch at 1000 agents. That means the problem is not "JFC cannot invoke 50+ agents." The gap is productization and lifecycle depth around that primitive: recursive child-session identity, fleet/daemon scaling, remote adversarial review, richer streaming lifecycle, and discoverable workflow/review surfaces.

## Things JFC already has and should not duplicate blindly

- JS workflow DSL: `agent()`, `parallel()`, `pipeline()`, `workflow()`, `phase()`, `log()`.
- Ultracode core mode: `/effort ultracode` and `ultracode` keyword handling already set xhigh plus standing Workflow guidance.
- Local prompt skills and agent skill preloading.
- Rich fixed built-in prompt seed with skills, dispatch, diagnostics, tool guidance, coding rules, safety rules, MCP metadata, and prompt-context descriptors.
- `StructuredOutput` tool validation for subagents/council.
- Workflow fanout limits: 16-way max concurrency and 1000-agent lifetime cap.
- Workflow resume: DB-backed append-only journal keyed by chained request hashes.
- Nested workflows: max depth 3 with budget propagation.
- Limited recursive Task tools inside subagents: depth limit 2.
- Structured subagent output: task-local schema and `StructuredOutput` tool validation.
- Worktree isolation and change-set preservation for task runs.
- Basic session resume/listing, worktree slash commands, themes, safe mode, plugin doctor, and per-session remote control.
- `AskUserQuestion` modal, brief-mode `SendUserMessage` guidance, notebook tools, and voice input basics.
- Basic planning/workflow skills, task rows, toasts, spinner tips, `/worktree`, `/sessions`, and config/theme/help commands.
- Memory storage already has user/feedback/project/reference types plus private/team scopes; parity work should focus on prompt-policy packaging and controls, not re-adding the taxonomy.
- JFC already vendors older Claude skill material (`claude-2.1.167`), including design-sync scripts, verify/run skills, Claude API docs, and the fewer-permission-prompts Markdown; parity work should make that corpus versioned and updateable.
- Agent-launch descriptors and process-bridge launchers for plugin-owned agents.
- Bounty/solver/validator primitives, though not yet packaged like `ultrareview`.
